Glass Skull local-only architecture
==================================

Glass Skull is a local-only application. Models are run locally. llama.cpp is
the execution backend. The app does not call remote model services and does not
download model weights.

Local execution backend
-----------------------

The normal runtime is a local llama.cpp server using a GGUF model selected by
the user. Glass Skull stores the server URL, model alias, GGUF path, local
llama-server path, and local llama-cvector-generator path in Streamlit session
state through the Settings tab.

The managed local source lives under:

    managed/llama.cpp-glass/

The patch record lives under:

    patches/llama.cpp-glass/0001-glass-skull-per-request-steering.patch

Local environment patches are expected. They should be documented by keeping
the patch file current, recording the managed llama.cpp commit, and verifying
that the managed checkout can be reversed or rebuilt from that patch record.

Core workflow
-------------

1. A user enters a prompt or batch of prompts in the Run tab.
2. Glass Skull sends chat requests to the configured local llama.cpp server.
3. When tracing is enabled, Glass Skull also calls the local
   /glass-skull/trace endpoint.
4. The trace payload is normalized into run-artifact rows preserving run id,
   batch id, prompt id, token id/token index, label, layer, stream, token text,
   activation norm, top dimensions, optional vectors, trace source, and
   unavailable-reason diagnostics.
5. The app stores a local artifact with prompts, outputs, trace rows, behavior
   scores, batch metadata, and error state.
6. The Map tab turns the artifact into a canvas payload and visualizes local LLM
   activity.
7. The Steer tab prepares local control sets, generates GGUF control vectors,
   and shows llama.cpp launch commands for steered runs.
8. The Timeline tab compares behavior scores across local runs.

Activation map contract
-----------------------

Layers represent local model blocks from GGUF metadata or the highest traced
layer returned by llama.cpp.

The activation map is a guitar-neck/fret visualization of prompt-token activity
through layers. It must preserve run id, batch id, prompt id, token id, and layer
through the whole pipeline. It must never merge unrelated prompts or tokens into
one pretty path.

Nodes represent activation locations derived from real trace rows. With full
vectors, nodes and edges are vector-backed. With top-dimension rows, nodes and
edges are explicitly labeled as top-k approximations. With scalar-only rows, the
renderer shows aggregate heatmap context only. It does not invent prompt paths.

Activation paths are per prompt and per selected token. Adjacent layers may be
connected only when the source rows share the same prompt id and token id. The
default mode is:

- single_prompt when one prompt has available trace rows
- batch_overlay when multiple prompts have available trace rows
- aggregate_heatmap only when selected by the user or forced by aggregate-only
  or scalar-only backend data

In batch_overlay, each prompt gets its own path for the selected token. Fifteen
unrelated prompts must show fifteen prompt paths, not one averaged or smeared
trajectory.

aggregate_heatmap is not a path mode. It shows activation density/cloud data and
must not draw a single connected trajectory unless a future centroid feature is
explicitly labeled as a centroid.

compare_prompts shows deltas between selected prompts. It does not combine
prompt activations into a shared path.

Heatmaps summarize activation density by layer/node/prompt context. They are
diagnostic summaries, not causal proof and not prompt trajectories.

Tooltips and diagnostics expose the exact selected source data: run id, batch id,
prompt id, token id/token text, layer, node id, cluster id, activation,
normalized activation, source fields, confidence, mode, edge method, and whether
the visual element is real or approximate. If the backend trace is already
aggregated, the UI must say that per-prompt paths are unavailable and render
aggregate heatmap only.

Intentionally not supported
---------------------------

The following systems are intentionally not supported in this repository:

- Hugging Face
- TransformerLens
- remote model providers
- bundled model download tooling

Models must be obtained and managed outside this repository, then pointed at
through a local GGUF path and local llama.cpp server configuration.
